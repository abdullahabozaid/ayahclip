## 2. What shipped this session (all live on ayahclip.com)

Each has a commit; all have unit tests and were verified in a real browser.

1. **Blurry preview text fixed** (`6988a20`). The studio preview canvas kept a
   full 1080×1920 backing squeezed into a ~292px box; the compositor minified it
   in one pass, blurring Arabic diacritics + English while export stayed crisp.
   Fix: backing store = on-screen size × `devicePixelRatio` (capped at export
   resolution) via a `ctx.setTransform` scale — `drawScene` still composes in
   export coordinates, so the single render path and preview=export parity hold.
   Files: `src/components/StudioPreview.tsx` (drawRef), `e2e/export-format-parity.spec.ts`.

2. **Saved-clip data corruption fixed** (`5cf1567`). Opening saved clip A then
   composing a new clip via the verse picker left the stale `projectId` in the
   store, so the 2s autosave overwrote A's saved record with the new clip's
   audio/captions. Fixes: `/surah/[id]` calls `beginNewProject()` on mount;
   `saveNow` refuses to overwrite a record whose `surahId` differs (mints a new
   id); media-dedup refs scoped per project id; `restoreProject` rebuilt on a
   shared `restoreSettingsDefaults()` so a record missing a key restores that
   feature OFF instead of inheriting the previous clip's value (~20 settings).
   Files: `src/lib/store.ts`, `src/app/surah/[id]/page.tsx`, `src/app/studio/page.tsx`.

3. **Bulk per-clip media persistence** (`5cf1567`/`490a303`). A bulk clip had no
   persisted per-clip look, so edits reverted on exit/return. Now
   `candidate.styleOverride` (durable-media-aware; blob URLs stripped) is saved on
   studio exit/sibling-nav/unmount and re-applied last on rebuild. Files:
   `src/lib/bulk-clips.ts`, `bulk-studio.ts`, `style-snapshot.ts`,
   `components/bulk/BulkCreateWorkspace.tsx`.

4. **720p portrait-video import fixed** (`c6fadbd`). HD import filtered YouTube
   formats by `[height<=720]`; a portrait "720p" video is 720 *wide* (e.g.
   720×808), so its height exceeds 720 and the real 720p was rejected → silently
   fell to 480p. Fix: cap by yt-dlp's `res` sort field (min dimension = shorter
   side = YouTube's "p" label). Also better error messages (too-large, bot-check).
   Files: `src/lib/social-download-jobs.ts`.

5. **Bulk multi-surah segmentation + keep-tab-open notices** (`490a303`). New
   "Recognised surahs" overview groups clips by surah (name, ayah range, clip
   count) with jump-to; helper `groupCandidatesBySurah` in `bulk-clips.ts`.
   Keep-tab-open notices during analysis + rendering (bulk runs client-side).

6. **Bulk "0 drafts" fixed + caption on/off** (`627b059`) — the big one, see §3.

Also fixed pre-existing branch breakage that had CI red: an ambiguous
"Prayer rows" e2e locator, a sub-12px `/styles` caption, and 4 stale reciter
e2e specs not updated for the popular-first picker.

## 3. Bulk Create recognition — how it works and what was fixed

**The pipeline (all CLIENT-SIDE except the download):**
- Download (yt-dlp) — SERVER (`src/lib/social-download-jobs.ts`, `/api/social-download`).
- ASR recognition + forced alignment — CLIENT, browser main thread
  (`src/lib/asr.ts` FastConformer Arabic CTC via onnxruntime-web; orchestrated by
  `src/lib/bulk-recognition.ts` → `quran-recognition.ts` → `verse-match.ts` +
  `forced-align.ts`).
- MP4 render — CLIENT (canvas + mp4-muxer, `src/lib/export.ts`).
- Checkpoints — IndexedDB (`src/lib/bulk-jobs.ts`).
- **There is NO server-side ASR/render worker.** Bulk needs the tab open.

**The "0 verse-complete drafts, N ambiguous windows withheld" bug (FIXED `627b059`):**
Recognition splits the source into 4-min windows. `recognizeQuranPassage` returns
`matched` / `ambiguous` / `none`. The `ambiguous` outcome ALREADY carries
fully-built, corpus-aligned candidate ranges (`outcome.candidates`), but
`bulk-recognition.ts` was discarding them — treating `ambiguous` identically to
`none`. On real recitations almost every window lands `ambiguous`, so the feature
produced nothing.

Fix: surface the top candidate of each ambiguous window as a **LOW-confidence
review draft** — built UNAPPROVED, `confidence: "low"`, "REVIEW RANGE" badge, with
messaging "N to review". The Arabic shown is still verified corpus text; only the
range attribution is uncertain (hence the review gate). Confident matches still
auto-approve; truly-empty (`none`) windows are still withheld. **Recognition
accuracy was NOT changed** — only what bulk does with the outcome.
- Added `"low"` to `BulkDetectedAyah.confidence` union + `confidenceRank`
  (`low: -1`). `buildVerseCompleteCandidates` auto-approves only all-confident
  clips (`isConfidentCandidate`). UI badge + counts in `BulkCreateWorkspace.tsx`.

**Verified live 2026-07-20:** importing a 7-min section of an Al-Furqan recitation
that previously gave 0 drafts now yields **7 reviewable drafts** ("7 to review"),
correctly recognised as Al-Furqan with verified Arabic + translation, approvable →
"Render approved (1)" enables.

**Caption on/off (SHIPPED `627b059`):** job-level `captionMode: "captions" |
"original"` (default captions). "Keep original (no captions)" sets
`arabicEnabled=false` + `translationEnabled=false` on rebuild so source clips that
already have burned-in captions aren't double-captioned. UI toggle in the bulk
Captions fieldset. Files: `bulk-jobs.ts`, `bulk-studio.ts`, `BulkCreateWorkspace.tsx`.

## 4. WHY recognition marks so much "review" (the real quality gap — 40%→)

The review-draft fix makes bulk *usable*, but almost everything comes back
low-confidence because of these, in order of leverage (all in `plans/008`):

1. **`hasCompetingRecognitionWindow` misfires** (`verse-match.ts:612-627`): it
   flags any pause-window candidate that is a *non-overlapping same-surah* range
   (`ayahStart > primary.ayahEnd`) as "competing", even though that's normal in a
   4-min continuous recitation. This forces confident single-surah windows into
   `ambiguous`. **Fixing this (behind the recognition benchmark) is the #1 next
   task** — it should convert most "review" drafts into confident auto-approved
   ones.
2. **4-min windows** force a single contiguous range over 5-15 ayahs; greedy-CTC
   errors accumulate → `match.score` below the strict `0.84/0.9` floors →
   `low` → withheld.
3. **Generic Arabic ASR** (greedy CTC, no LM/beam) degrades on melodic tajweed.
4. **No cross-window merge** — the plan specifies merging a continuous
   surah/ayah sequence across windows; it doesn't exist.

Research finding (see `plans/008`): the winning approach is **closed-vocabulary
forced alignment** (identify passage → align the KNOWN ayah text with a
passage-filtered dictionary), not open ASR — quran-align hits ~73ms boundary
accuracy this way. Also: Quran-tuned models (`tarteel-ai/whisper-base-ar-quran`),
Opus-Clip-style clip scoring + review grid, burned-in caption OCR detection.

## 5. Hard invariants — do NOT break these

- **Quran text integrity is sacred.** Mis-rendered Arabic (waqf marks wrapping,
  dropped diacritics, wrong verse) is a fatal bug. Never present an unverified
  range as confident/approved. Any change to `verse-match.ts` /
  `quran-recognition.ts` / `render-core.ts` `sliceQcfForDisplay` is Quran-critical
  and must be benchmark-gated (`npm run test:recognition`, `test:detection`,
  `test:alignment`).
- **Preview must equal export.** ALL frame composition goes through `drawScene`
  in `src/lib/render-core.ts` at export resolution. Never add a second render
  path. The blur fix keeps this — it scales the transform, not the composition.
- **Clip state isolation** (see §2.2): the verse picker must reset project
  identity; autosave must not overwrite a different clip's record; `restoreProject`
  must fall back via `restoreSettingsDefaults()`. Regressing this silently
  corrupts saved clips.
- **Bulk per-clip look** persists via `candidate.styleOverride`, applied last on
  rebuild in BOTH `bulk-studio.ts` and `BulkCreateWorkspace` render path.
